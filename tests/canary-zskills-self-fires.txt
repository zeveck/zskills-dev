## Why a recipe and not a test

Claude Code reads `.claude/settings.json` ONCE at session start. A test
that edits `settings.json` mid-run cannot reload the hook registry â€” only
a session restart does. The unit tests under `tests/` exercise the hook
script's pure-bash logic (input/output, JSON envelope, deny vs. allow
decisions); they cannot exercise the harness wiring. That is what this
recipe covers.

## Manual procedure

Run from a fresh Claude Code session (post-install / post-`/update-zskills`):

1. Pick any source skill, e.g. `skills/commit/SKILL.md`.
2. Edit ONE byte of the SKILL.md body (e.g. add a trailing newline to a
   prose paragraph) â€” DO NOT bump `metadata.version`.
3. Stage it: `git add skills/commit/SKILL.md`
4. Issue a bare commit (NOT through `/commit`):
   `git commit -m "test stale-skill-version hook"`
5. Expected: the harness renders a tool-deny envelope. The
   `permissionDecisionReason` field contains the
   `skill-version-stage-check.sh` STOP message verbatim, including the
   suggested bump command:
   `bash scripts/frontmatter-set.sh skills/commit/SKILL.md metadata.version "<today>+<hash>"`
6. The commit DOES NOT happen. `git status` shows the file still staged.
7. Run the suggested bump command, re-issue the commit â€” it succeeds.

## Failure mode if canary trips

If the manual recipe is run and the hook DOES NOT fire (commit succeeds
with a stale skill version), the most likely cause is that the consumer
session has not yet absorbed an `/update-zskills` after Phase 4's helper
ship. Re-run `/update-zskills`, restart the session, retry the recipe.

If the recipe fails AFTER a fresh `/update-zskills` + restart, that is a
real regression â€” file an issue with the failing skill, the
`.claude/settings.json` excerpt for `PreToolUse`, and the hook script
path on disk.
